Java 的错误定位与 logback 详解

也是奇了怪了，只要写代码，不管多简单的东西都得出点错，所以这里整理了一下遇到错误具体怎么排查，算是通用操作了

这里主要是针对 web 项目来说，刷题啥的出错主要出在逻辑上，我也一直在找一种可以 debug 大脑的方式，回头找到了一定分享出来，主要分为以下几步：

  1. 首先判断页面响应状态码：主要看问题在哪方面，当然这里的 ExceptionMessages 也会打印在 IDE 的控制台上
  2. 断点调试，这里包括客户端的 JS 调试，不过一般问题出在后端，所以 Debug 的几个快捷键要记熟
  3. 设置日志级别，这个在开发时候用的不多，但是项目上线后没有个日志怕是不太行，所以这里重点介绍 Java 的日志体系

至于一些具体错误的排查记录在 问题解决

判断响应码

发起一个请求失败通常是返回一个错误页，比如常见的 404 页面就表示没有找到相应请求，具体含义可以在 MDN 进行查看，这里列出两个常见的

  • 400(Bad Request)：请求参数有误；语义有误，当前请求无法被服务器理解，除非进行修改，否则客户端不应该重复提交这个请求
  • 404(Not Found)：请求失败，请求所希望得到的资源未被在服务器上发现。没有信息能够告诉用户这个状况到底是暂时的还是永久的。假如服务器知道情况的话，应当使用 410 状态码来告知旧资源因为某些内部的配置机制问题，已经永久的不可用，而且没有任何可以跳转的地址。404 这个状态码被广泛应用于当服务器不想揭示到底为何请求被拒绝或者没有其他适合的响应可用的情况下
  • 500(Internal Server Error)：服务器遇到了不知道如何处理的情况

至于开发时对于错误页是有默认配置的，如果不想使用，也可以在 /templates/error/ 下放置页面，命名可以使用具体的错误代码或是 4xx.html 这种形式表示所有客户端错误都由此页面响应，同理还有 5xx.html

但注意一点是开发时自定义的页面尽量也要对后端的错误信息进行一个反馈，方便了解异常的详细信息，比如加一句： <h2 th:text="${message}">!!!</h2>

断点调试

有些时候问题可能不仅是出现在后端，所以在定位时不详细的情况下，可以先对前端进行断点检查

前端调试

前端需要排查的错误有：请求地址、请求方式、请求参数、响应的数据，确定这些都没错之后就可以直接去后端找 bug 了

常见的解决思路：

  1. 用请求地址（url）和键值对直接进行访问，如果有正确的返回数据，则说明服务器端没有问题
  2. 通过打断点查找到底是请求的问题，还是响应的问题
    • 在请求上打断点，看请求地址和请求参数有没有问题（可以把请求地址和请求参数定义成变量的方式，方便查看）
    • 断点打在回调函数里（函数体的第一句话，而不是 function 关键字所在行），看后端响应回来的数据有没有问题

打断点方式：页面上单击 F12 --> 源 (Sources) --> 找到需要调试的源码文件 --> 脚本代码显示区域左边的数字上添加断点即可；或是在需要调试的地方加 debugger 关键字，代码运行到的时候会自动停下，进入调试模式

注意：有的数字行是灰色的，就是不可断点。有时候点击 15 行断点选中 14 行，这是因为浏览器真正执行的代码行不是你看到的那一行，可能是优化掉了或者经过某种转换。还有的情况是，添加某一行断点，会跳转到另一个页面并命中某一行，这个页面的背景色是黄色且文件名是 VM 开头。上述情况，大部分都是因为浏览器显示的源码版本，跟真正的源码文件不一致，只需要刷新一下页面，保证显示的源码是最新的即可

调试技巧：在浏览器中找不到想打断点的源码位置时可以在需要调试的那一行代码前加上 console.log(666) ，然后运行一次，到控制台查看结果输出，点击右边的链接会自动跳转到源码，这样就可以直接定位

后端调试

后端进行调试时最好配上专门的接口调试工具，最常用的 postman，比较常用的排查技巧：

  1. 在代码中间打印结果，适合对某一块代码不自信的情况下
  2. 打断点进入 debug 模式，也就是跟着代码走一遍 Debug 教程

日志打印

刚才说 Debug 是跟着代码走一遍，还是有点费时费力，所以可以通过日志将具体都干了什么打印出来

SpringBoot 中自带的日志组件有 Logback 、 SLF4J ，默认情况下，SpringBoot 会用 Logback 来记录日志，并用 INFO 级别输出到控制台

SLF4J 其存在主要就是简化日志输出的操作，没有 SLF4J 的情况下我们想打印日志，需要先定义一个使用当前类构造的日志对象： private final Logger logger = LoggerFactory.getLogger(thisClass.class) ，然后调用 logger.info(message)

如果使用 SLF4J 只需要在类上进行 @Slf4j 注解，然后就可以在任意位置使用 log.info(message)

现在回来观察一下启动时控制台输出的日志信息

不去关心具体内容我们发现其都有一个固定的格式：先打印时间，然后一个方括号包裹的内容，然后是 INFO …，其实我这不是完整的日志格式，完整的日志格式包括：

  • 时间日期：精确到毫秒
  • 日志级别：ERROR, WARN, INFO, DEBUG or TRACE
  • 进程 ID
  • 分隔符：— 标识实际日志的开始
  • 线程名：方括号括起来
  • Logger 名：通常使用源代码的类名
  • 日志内容

输出方式

日志级别从低到高分为 TRACE < DEBUG < INFO < WARN < ERROR < FATAL，如果设置为 WARN，则低于 WARN 的信息都不会输出。
SpringBoot 中默认配置 ERROR、WARN 和 INFO 级别的日志输出到控制台，当然也可以在配置文件中更改打印的级别： debug=true

使用 logging.level.packageName=debug 可以对某个包下的文件设置级别，可以 Mapper 接口都使用 debug 级别进行打印，方便进行错误检查

文件输出

输出到控制台固然简单，但阅读起来不方便，况且部署后的项目又没有 IDE 可以进行控制台查看，所以使用文件保存日志信息是很有必要的，在 application.properties 中设置 logging.file.path 或 logging.file.name 属性（二者同时使用时只有 logging.file 生效）

  • logging.file，设置文件，可以是绝对路径，也可以是相对路径，会在项目的当前路径下生成一个 xxx.log 日志文件
  • logging.path，设置目录，会在该目录下创建 spring.log 文件，并写入日志内容

默认日志文件的大小达到 10MB 时会切分一次，产生新的日志文件，默认级别为：ERROR、WARN、INFO。这样操作起来还是有局限性，比如我们想不同的隔离级别用不同的方式来配置，这样就需要外部文件了

日志服务一般都在 ApplicationContext 创建前就初始化了，它并不是必须通过 Spring 的配置文件控制。因此通过外部配置文件依然可以很好的支持日志控制和管理，logback 配置文件放在类路径下使用 logback-spring.xml 或 logback.xml 命名会被自动识别，使用 logback-spring.xml 的日志配置文件，SpringBoot 可以为它添加一些 SpringBoot 特有的配置项，日志外部配置文件的名字也是可以自定义的： logging.config=classpath:logging-config.xml

配置文件解释

首先看一下 xml 的标签结构

configuration：
前三个为标签属性，后两个是紧跟的从属性

  • scan：为 true 时，配置文件如果发生改变，将会被重新加载，默认值为 true。
  • scanPeriod：设置监测配置文件是否有修改的时间间隔，如果没有给出时间单位，默认单位是毫秒。当 scan 为 true 时，此属性生效，默认的间隔为 1 分钟
  • debug：为 true 时，将打印出 logback 内部日志信息，实时查看 logback 运行状态，默认值为 false
  • contextName：用于区分不同应用程序的记录，默认上下文名称为“default”，一旦设置，不能修改，可以通过 %contextName 来打印日志上下文名称
  • property：定义键值对，后续可以使用 ${} 进行引用

appender：

  • class：标签属性，用来指定哪种输出策略，常用就是控制台输出策略（ch.qos.logback.core. ConsoleAppender）和文件输出策略（ch.qos.logback.core.rolling. RollingFileAppender）
  • filter：定义拦截器和级别，ThresholdFilter 为系统定义的拦截器
  • encoder：对输出的每行日志进行格式规定
    • %d{HH: mm:ss. SSS}：日志输出时间
    • %thread：输出日志的进程名字，这在 Web 应用以及异步任务处理中很有用
    • %-5level：日志级别，并且使用 5 个字符靠左对齐
    • %logger{36}：日志输出者的名字
    • %msg：日志消息
    • %n：平台的换行符
  • rollingPolicy：配置滚动策略，常使用ch.qos.logback.core.rolling. TimeBasedRollingPolicy
    • fileNamePattern：日记拆分方式，使用 ${log.path}/logback.%d{yyyy-MM-dd}.log 把每一天的日志归档到一个文件中
    • maxHistory：表示只保留最近 X 天的日志，可以使用 %d{yyyy-MM-dd_HH-mm}来精确拆分
    • totalSizeCap：指定日志文件的上限大小达到后进行删除
      天数和大小生成冲突可以使用 SizeAndTimeBasedRollingPolicy 进行

root: 必选节点，用来指定最基础的日志输出级别，只有一个 level 属性，可以包含零个或多个元素，标识这个 appender 将会添加到这个 logger

logger：
用来设置某一个包或者具体的某一个类的日志打印级别、以及指定。有一个 name 属性，一个可选的 level 和一个可选的 addtivity 属性

  • name: 用来指定受此 logger 约束的某一个包或者具体的某一个类
  • level: 用来设置打印级别，未设置此属性，那么当前 logger 将会继承上级的级别
  • addtivity: 是否向上级 logger 传递打印信息，默认是 true
